# Todo Reminder 項目評估報告

> **日期**：2026-04-28
> **版本**：v0.2.0 (commit `2b0d839` 之後的工作中分支)
> **作者視角**：項目經理 / 技術負責人
> **目的**：盤點現況問題、優化機會、擴展可能性，產出可執行的優先排序

---

## 0. 摘要 (Executive Summary)

Todo Reminder 從 0 到 v0.2.0 在約一週內完成了 **MVP + 一輪 polish + Fantastical-style 重設計 + 天氣 / 農曆 / 節假日整合**。應用目前狀態：

- **可日常使用**：MSI 安裝、Windows 通知、托盤常駐、開機自啟皆 OK
- **資料層穩定**：tasks / subtasks / tags / reminders / activity_log / settings 6 表 schema，已過 migration v2
- **視覺整體成形**：Fantastical-like 月曆 + Linear-lite 看板，但**未到「精緻」門檻**

但仍有顯著不完整之處，分四類：

| 類別 | 嚴重度 | 估算總工時 |
|---|---|---|
| 視覺一致性 / 設計細節 | 中 | 4-8h |
| 功能缺口（已 schema 未 UI） | 中 | 12-20h |
| 效能 / 穩定性風險 | 低-中 | 6-10h |
| 工程基礎 (測試 / CI / 文件) | 低 | 8-16h |

**三大立即優先**：
1. **子任務 UI** —— `subtasks` 表 schema 與 API 已備，但任務編輯時無法新增 / 編輯子項目
2. **拖拽改期 / 跨欄移動** —— 看板沒有 drag-drop，月曆無法拖事件改日
3. **多提醒 + 提醒於休眠喚醒後補發** —— 目前每任務只能綁一個提醒，且 tokio interval 不感知系統 suspend

## 1. 樣式 / 視覺問題

### 1.1 仍有改進空間

**1.1.1 設計語言一致性**
- TitleBar 的 view tab 是「嵌入滑塊」風格（凹陷容器 + 浮起 active），但其他地方的切換器（如 `SettingsDialog` 內的主題切換）用的是 outline pill 風格。
- 任務優先級顏色：Kanban Column 用 `--col-*` 系列，TaskCard 用 `--prio-*`，月曆膠囊用獨立的 HSL 字面量。**三套體系**未統一。
- shadow 系統有 `shadow-card`、`shadow-card-hover`、`shadow-float` 三層，但實際使用稀疏，部分元件直接寫了 arbitrary `shadow-[0_2px_4px_...]`。

**建議**：建立 `src/lib/design-tokens.ts` 收斂顏色 + shadow + 間距 token，TS 端引用而非到處 inline。

**1.1.2 字體層級弱**
- 月曆主區無大標題（為了 Fantastical-like 簡潔），但這也意味整個畫面**沒有焦點字級**。眼睛沒有抓地方。
- TitleBar 的 `Todo Reminder` (13px) vs sidebar mini cal `2026 4月` (14px) vs 內文小字 (10–12px)：層級壓縮在 10–14px 區間，缺乏對比。

**建議**：保留主月曆無標題，但 sidebar 月曆標題拉大到 16–18px，建立明顯的視覺 anchor。

**1.1.3 dark mode 與 sidebar 衝突**
- Sidebar **永遠**深色（為了 Fantastical 對比效果），但用戶切到 dark mode 時，主區也是深色，sidebar 跟主區色差**幾乎消失**。設計初衷只在 light mode 成立。
- 需要在 dark mode 時，把 sidebar 改成「比主區更深一階」或「比主區亮一階」，保持層次。

**建議**：sidebar 在 dark mode 下用 `bg-[hsl(225_22%_5%)]`（比主區的 `224 18% 8%` 再深 3%）。

**1.1.4 月曆視覺密度問題**
- 視窗 1180×760 下，月曆 6 行×7 列 = 42 格，每格 ~92×135px。在當前任務量稀疏的情況下，看起來很空。
- 任務多時會超 4 條 + 顯示「還有 N 項」，缺少 hover/click 展開該日全部事件的方式。

**建議**：點日期格 → 側邊滑出當日完整事件列表（類似 Outlook 月檢視 + 日詳情 panel）。

**1.1.5 weather icon 過小**
- 主月曆每格的天氣 icon 是 12×12px，幾乎看不清是雲還是雨。
- Sidebar upcoming 的 12×12px 也偏小。

**建議**：放大到 14–16px。

### 1.2 已知小 bug

| ID | 描述 | 影響 |
|---|---|---|
| S-1 | Sidebar 內 `text-foreground` 因為套了 `dark` class，在 light theme 下強制反轉，但有些子元件用了硬編碼顏色（如 weather card 的 high 溫度），不會跟隨 | 微 |
| S-2 | TaskForm 的 datetime picker popover 在 viewport 邊緣會 auto-flip，但 sidebar 模式下 popover 仍可能溢出 sidebar 右邊界 | 微 |
| S-3 | 月曆 mini month 「當前週」高亮在 light theme 下也存在，但因為 sidebar 是深色，相對對比正常；如果未來 sidebar 改主題感知就需重審 | 待重審 |
| S-4 | `lunar-typescript` 在簡體 region 直接使用其原始輸出（如 `劳动节`），TW/HK region 走自家映射表，但**節氣只做了 5 個 s2t 映射**，其他簡繁同形的就直接顯示 | 低 |
| S-5 | 上次自截圖看到 sidebar mini cal 沒有顯示「今天」之外的多色點 — 可能是測試任務不夠多 | 等實測 |

### 1.3 設計系統未建

目前沒有：
- 設計 tokens 集中管理檔
- 元件庫文件化（哪些 props、變體、用法）
- 設計 review 流程

**建議優先級**：低（單人專案，過早抽象 = 浪費）。但如果加第二個開發者就要做。

## 2. 功能缺口

### 2.1 已有 schema 但無 UI

| 功能 | 後端狀態 | 前端 API | 前端 UI | 工時估算 |
|---|---|---|---|---|
| **子任務** | ✓ 表 + index | ✓ `src/api/subtasks.ts` | ✗ | 6h |
| **標籤** | ✓ 表 + 多對多 | ✓ `src/api/tags.ts` | ✗ | 8h |
| **多提醒** | ✓ schema 支援 | △ 只 upsert 主提醒 | ✗ | 4h |
| **活動日誌** | ✓ 表 | ✗ | ✗ | 6h |

**子任務**：TaskForm 內加一個區段，可新增/勾選/拖拽排序子項。每個 task 顯示子任務完成度（如 3/5）。

**標籤**：TaskForm 加多選器，看板 / 月曆顯示彩色標籤 chip。設定頁可管理標籤（增刪改色）。

**多提醒**：TaskForm 的「提醒」section 改為列表，可加多個（如「截止前 1 天 + 截止前 1 小時」雙提醒）。

**活動日誌**：新增「活動」面板，列出最近的建立 / 更新 / 完成 / 刪除動作。對「我做了什麼」回顧很有用。

### 2.2 plan.md 內標 v0.2 但未做

- ❌ **拖拽** 任務 between columns / between days
- ❌ **快速新增全局快捷鍵**（不在 app 內）—— `Ctrl+Shift+T` 全局喚出迷你輸入框，無論 app 在哪
- ❌ **週期任務**（每天 / 每週 / 每月重複）
- ❌ **任務模板**（常用任務一鍵生成）
- ❌ **匯入匯出** JSON / Markdown
- ❌ **通知 action buttons**（toast 上有「完成 / 稍後 5 分鐘」直接動作）

### 2.3 已實作但有限制

**2.3.1 自然語言時間解析**
- 用 `chrono-node`，中文支援有限
- 「下個月十五號開會」可能解析失敗
- **改善**：fallback 到簡單關鍵字匹配（「明天」「後天」「下週X」），或接 `chinese-lunar-parser` 類庫

**2.3.2 提醒 scheduler**
- Tokio `interval(30s)` 在系統 sleep 時不 tick
- 醒來後不會主動補發 sleep 期間錯過的提醒（嚴格說會：每次 tick 查 `remind_at <= now AND fired = 0`，所以醒來的下一個 tick 會掃到，但會延遲最多 30s）
- **完善**：監聽 Windows power resume 事件，主動補一次 fire_due

**2.3.3 月曆視圖只有月**
- 沒有 day / week / year 視圖
- Fantastical 全有
- **建議優先級**：低，月視圖對 todo 已足

**2.3.4 跨日任務在月曆顯示**
- 目前每天獨立渲染一個 pill（`測試`「測試」「測試」連續四格）
- Fantastical 是**橫跨多格的連續色條**，視覺上更有連續感
- **實作複雜度**：高（需 lane 分配 + CSS column span）
- **建議**：v0.4 再做

**2.3.5 地區偵測 + 簡繁混雜**
- `auto` 預設偵測，依 locale + timezone
- 但簡體用戶看 TW 內容（國定假日）會奇怪，反之亦然
- 目前只把節氣 5 個簡繁字做 s2t（`谷雨`→`穀雨` 等）
- 完整繁簡轉換需要 OpenCC 級別的庫

**2.3.6 天氣**
- 只支援 IP 定位，沒有手動覆寫（如「我在台北，但 IP 在新加坡 VPN」就錯）
- 沒有「重新整理天氣」按鈕
- 失敗時無錯誤訊息，只是不顯示

**2.3.7 設定頁過於簡陋**
- 目前只有：主題、區域、開機自啟
- 缺：時區、第一天是週一/週日、預設提醒偏移、通知聲音選擇、城市覆寫、語言（繁/簡 UI 切換）

## 3. 效能與穩定性

### 3.1 效能

**3.1.1 任務量擴展性未驗證**
- 目前所有 tasks 一次 `loadAll()` 拉進 zustand store
- 1k 任務以下流暢，10k+ 沒測過
- 月曆每格 `taskWindowOverlaps` 對所有 tasks 線性掃描 = O(N × 42)
- **改善路徑**：DB 端 index `start_at`、`due_at`、`status`；前端按月查 `WHERE due_at BETWEEN ?,? OR start_at BETWEEN ?,? OR (...)` 只拉可見範圍的任務

**3.1.2 月曆每月切換無快取**
- 每次切月，`tasksByDay` useMemo 重算
- 對當前任務量無影響，但隨任務增多會有感

**3.1.3 天氣每次開 app 都重抓**
- 進程啟動 → `useEffect` → `loadWeather()` → IP geo + open-meteo
- TTL 1 小時，但這是 in-memory，重啟就清空
- **改善**：weather 結果寫進 settings 表（key=`weather_cache_v1`）+ 帶時間戳，啟動時讀 cache，背景 stale 才重抓

**3.1.4 sqlx_query DEBUG log 過多**
- Rust 側 log level 預設 DEBUG，每個 SQL 都印一行（看截圖 b1k6lu5n9.output 有 100+ 行 SQL log）
- Release 應改為 INFO，dev 可保 DEBUG

### 3.2 邊界條件

**3.2.1 時區**
- DB 全用 Unix 秒（UTC）
- UI 轉 local 用 `new Date(sec * 1000)`
- 如果用戶旅行跨時區，**已建任務的「截止時間」會跟著系統時區變動**——這是 todo 的常見爭議點
- 不一定是 bug，但要文件說清楚

**3.2.2 系統時間被改動**
- 用戶手動改系統時間，Tokio interval 可能怪異
- 影響 fired reminder 的判斷
- **解決**：加上 sanity check（now 不能比上次 tick 倒退）

**3.2.3 DB 損毀**
- 沒有自動備份
- 沒有匯出
- 用戶誤刪 / DB 檔案損壞 = 全部資料丟失
- **建議**：每天自動備份到 `%APPDATA%\com.lenvovo.todoreminder\backups\todo-YYYY-MM-DD.db`，保留 7 天

**3.2.4 同時兩個 app 實例**
- `tauri-plugin-single-instance` 已裝，但測試覆蓋率？
- 如果兩個實例競爭同一 DB（如殘留的 zombie 實例），SQLite WAL 應該安全，但提醒可能重複觸發

### 3.3 資料完整性

- `reminders.fired` 設為 1 後不會 reset，即使任務改了 due_at
- 改 due_at 後，**舊提醒仍標 fired = 1**，新提醒在 setReminder 時 upsert 新值，但「之前已 fired」的歷史紀錄留著（`fired = 1` 永遠 true）
- 看起來是正確的（避免重複發送），但 reminder 表會永遠長大，沒有清理機制
- **建議**：每月清一次 `WHERE fired = 1 AND remind_at < (now - 30 days)`

## 4. 程式碼品質

### 4.1 一致性

**4.1.1 顏色寫法**
- HSL 變數 `hsl(var(--primary))` ✓ 主流
- HSL 字面 `hsl(0_72%_52%)` ← 月曆「今天紅」、weather tone 等用了大量字面
- Hex 沒看到（好）

**整理建議**：把字面 HSL 抽到 `globals.css` `--cal-today`、`--weather-sunny`、`--weather-rain` 等 token，元件只用 `var(--cal-today)`。

**4.1.2 元件 props 命名**
- `onOpenChange` (Dialog 風格 from radix)
- `onOpenTask` (CalendarMonth)
- `onOpen` (TaskCard)
- 都是 callback 但命名不統一 —— 建議都改 `onTask*` 動作命名（`onTaskOpen`, `onTaskToggleDone`）

**4.1.3 store 命名**
- `useTaskStore` (action: `loadAll`, `add`, `patch`, `complete`, `remove`)
- `useUIStore` (`setViewMode`, `setSearchQuery`)
- `useSettingsStore` (`setTheme`, `setRegion`)
- `useToastStore` (`push`, `dismiss`)
- `useWeatherStore` (`load`)
- 部分 store 用 verb (push) 部分用 set+name —— 不統一

### 4.2 重複代碼

**4.2.1 priority → color 重複**
- `priorityDotClass`, `priorityPillClass`, `priorityBarClass` 在 TaskCard、CalendarMonth 各自定義
- 應該抽到 `src/lib/priority.ts`：
  ```ts
  export function priorityDotClass(p: Priority): string;
  export function priorityFillClass(p: Priority): string;
  export function priorityStrokeClass(p: Priority): string;
  ```

**4.2.2 weather 描述映射**
- `describeWeather` 已抽（lib/weather.ts），但 tone class 用 record 內聯—— OK，可以再 polish

**4.2.3 任務排序邏輯**
- `selectToday`, `selectThisWeek`, `selectBacklog`, `selectDone7d` 各自過濾
- `tasksByDay`、`UpcomingList` 也各自排序
- 排序函式重複寫了 3 次（done 沉底 + 優先級高的在前）
- **抽**：`src/lib/sort.ts` 集中所有任務排序函數

### 4.3 型別安全

**4.3.1 status 字串**
- `TaskStatus = 'todo' | 'doing' | 'done' | 'cancelled'`
- 但代碼裡有 `t.status.includes('done')` 這種寬鬆檢查（雖然意圖是 done，但 includes 在 'doing' 也會 false-positive 到 'done'... wait, includes 是 string.includes，'doing'.includes('done') 是 false，OK）
- 整體型別 OK

**4.3.2 lucide-react icon 型別**
- `LucideIcon` 用於 ComponentType 的 column 配置 — 有就好

**4.3.3 zustand store**
- 沒有用 immer，直接 spread —— 對小 state 沒事，但深層巢狀會繁瑣

### 4.4 死代碼

掃描出未使用：

- `formatDue` 在 datetime.ts 還在但 TaskCard 改用 `formatSchedule` 後可能不再呼叫——查一下

## 5. UX 細節

### 5.1 任務操作

**5.1.1 缺 undo**
- 刪除任務後沒有 undo 機會
- toast 顯示「任務已刪除」但不能撤回
- **改善**：toast 加 `[復原]` 按鈕，5 秒內可救（DB 端先軟刪除 → 確認後硬刪）

**5.1.2 缺批次操作**
- 沒法 select 多個任務同時：完成、刪除、改優先級、改截止
- **建議**：看板支援 Shift-click 多選

**5.1.3 從 TaskCard 直接改截止**
- 目前要點開 Dialog 才能改 due_at
- Fantastical 可右鍵 menu / 拖拽改
- **改善**：右鍵 TaskCard → context menu (改截止 / 改優先級 / 完成 / 刪除)

**5.1.4 子任務 checkbox 從卡片直接打勾**
- 子任務 UI 沒實作（見 2.1）

### 5.2 鍵盤

**5.2.1 已有**
- `Ctrl+N` 開進階表單 ✓
- `Ctrl+F` 聚焦搜尋 ✓
- `Esc` 關 Dialog ✓

**5.2.2 待加**
- `Ctrl+J` 切換看板 / 月曆
- `Ctrl+,` 開設定（macOS 慣例）
- `Enter` 在搜尋框 → 跳到第一個結果並打開
- `↑↓` 在搜尋結果列表中移動
- `Ctrl+1/2/3` 切換 Kanban 欄
- 月曆：`PgUp/PgDn` 上下個月

### 5.3 觸控 / 無障礙

**5.3.1 觸控**
- Tauri 桌面 app，但 Windows 11 觸控筆 / 觸控螢幕仍存在
- 任務膠囊小（11px 字 + 11.5px 容器）對觸控不友善
- 拖拽完全沒做

**5.3.2 ARIA**
- TaskCard 的 checkbox 有 aria-label ✓
- View tab 有 `aria-pressed` ✓
- 月曆格子是 `<div>`，沒有 role 也沒鍵盤可達 ✗
- **改善**：月曆格子可 Tab 進入，方向鍵導航

**5.3.3 高對比模式**
- Windows 高對比模式（Settings → Accessibility）下，CSS 變數會被 forced-colors 覆寫
- 沒測過，可能某些自訂顏色會被忽略導致看不清
- **建議**：加 `@media (forced-colors: active)` 規則保底

### 5.4 錯誤回饋

**5.4.1 Toast 系統**
- 現有 info / success / error 三類 ✓
- 錯誤訊息常常是 raw error string（如 sqlx error）—— 對用戶不友善
- **改善**：API 層 wrap 錯誤成用戶可讀文案

**5.4.2 DB 寫入失敗**
- 前端 catch 後 toast，但 store 狀態已先樂觀更新——失敗回滾邏輯沒做
- **改善**：每個 mutation 在失敗時 reload 對應子集

**5.4.3 weather 失敗無提示**
- 上面 2.3.6 提過
